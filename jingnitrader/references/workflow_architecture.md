## 阶段状态机

### 基础流程

```
[IDLE]
    │
    │ 用户输入
    ▼
[DATA] ──→ [FACTOR] ──→ [MODEL] ──→ [BACKTEST] ──→ [PORTFOLIO] ──→ [EXECUTION] ──→ [REPORT]
```

### 分支逻辑

#### 数据获取阶段 (DATA)

```
[DATA]
    │
    ├── 成功 → [FACTOR]
    │
    └── 失败 → [REPORT] (生成数据错误报告)
```

#### 因子构建阶段 (FACTOR)

```
[FACTOR]
    │
    ├── 成功 → [MODEL]
    │
    └── 失败 → [REPORT] (生成因子错误报告)
```

#### 模型训练阶段 (MODEL)

```
[MODEL]
    │
    ├── 成功 → [BACKTEST]
    │
    ├── 失败 → [FACTOR] (返回因子阶段重新调整)
    │
    └── 过拟合 → [FACTOR] (返回因子阶段优化)
```

#### 回测验证阶段 (BACKTEST)

```
[BACKTEST]
    │
    ├── 成功 → [PORTFOLIO]
    │
    ├── 失败 → [MODEL] (返回模型阶段调整)
    │
    └── 性能不佳 → [FACTOR] (返回因子阶段重新构建)
```

#### 组合优化阶段 (PORTFOLIO)

```
[PORTFOLIO]
    │
    ├── 成功 → [EXECUTION] 或 [REPORT]
    │
    └── 失败 → [REPORT] (生成组合优化报告)
```

#### 执行交易阶段 (EXECUTION)

```
[EXECUTION]
    │
    ├── 成功 → [REPORT]
    │
    └── 失败 → [REPORT] (生成交易错误报告)
```

## 核心组件

### 1. 意图解析器 (Intent Parser)

**职责：**
- 解析用户自然语言输入
- 提取关键参数（股票池、时间范围、策略类型）
- 确定目标阶段序列

**输入：**
- 用户自然语言描述

**输出：**
- 初始化的 Context 对象

### 2. 阶段调度器 (Stage Scheduler)

**职责：**
- 管理阶段执行顺序
- 处理阶段间的依赖关系
- 支持断点续跑

**输入：**
- Context 对象（包含 target_stages）

**输出：**
- 执行结果摘要

### 3. 产物校验器 (Artifact Validator)

**职责：**
- 验证每个阶段的产物完整性
- 检查产物文件是否存在
- 验证产物数据的基本合理性

**输入：**
- 阶段名称
- 产物文件路径

**输出：**
- 校验结果（通过/失败）

### 4. 错误处理器 (Error Handler)

**职责：**
- 捕获各阶段的异常
- 记录错误信息到 Context
- 提供错误恢复建议

**输入：**
- 异常对象
- 阶段名称

**输出：**
- 错误记录

## 数据流

### 主数据流

```
用户输入 → 意图解析 → Context 初始化 → 阶段调度 → 子 Skill 调用 → 产物生成 → 状态更新 → 下一阶段
```

### 产物传递

```
DATA产出物 → FACTOR输入 → FACTOR产出物 → MODEL输入 → MODEL产出物 → BACKTEST输入 → ...
```

### 状态传递

```
Context (task_id, user_intent) 
    ↓
Context (添加: target_stages, stock_pool, dates)
    ↓
Context (添加: artifacts[DATA], metadata[DATA])
    ↓
Context (添加: artifacts[FACTOR], metadata[FACTOR])
    ↓
...
```

## 断点续跑机制

### 检测点

每个阶段执行完成后自动保存检查点：

```python
# 检查点保存
checkpoint = {
    "task_id": ctx.task_id,
    "current_stage": stage,
    "completed_stages": completed_stages,
    "artifacts": ctx.artifacts,
    "metadata": ctx.metadata,
    "errors": ctx.errors
}
save_checkpoint(checkpoint)
```

### 恢复流程

```
启动 → 检查检查点是否存在 
    │
    ├── 存在 → 加载检查点 → 从断点继续执行
    │
    └── 不存在 → 解析新任务 → 从头开始执行
```

## 扩展性设计

### 添加新阶段

1. 在 `STAGES` 列表中添加新阶段
2. 在 `STAGE_ORDER` 中添加顺序映射
3. 在 `SKILL_MODULES` 中添加模块映射
4. 在 `EXPECTED_ARTIFACTS` 中添加产物定义
5. 实现对应的子 Skill

### 添加新后端

1. 在 `config.py` 中添加后端配置
2. 在对应的适配器目录创建新的适配器类
3. 实现标准接口方法
4. 在配置中注册新后端

### 添加新数据源

1. 在 `a-share-data-engine/adapters/` 创建适配器
2. 继承 `BaseDataProvider` 类
3. 实现 `fetch_daily()` 和 `fetch_minute()` 方法
4. 在 `config.py` 中注册新数据源

## 性能优化

### 并行处理

某些独立阶段可以并行执行：

```python
# 可以并行的阶段
parallel_stages = [
    ["DATA"],  # 并行获取多个数据源
    ["FACTOR"],  # 并行计算多个因子
]
```

### 缓存策略

- **产物缓存**：已完成阶段的产物直接使用
- **模型缓存**：训练好的模型保存到文件
- **数据缓存**：常用数据保存在本地

### 增量更新

```python
# 仅更新最新数据
if ctx.metadata.get("last_update"):
    start_date = ctx.metadata["last_update"]
```

## 监控和日志

### 日志级别

- **DEBUG**：详细执行信息
- **INFO**：一般执行信息
- **WARNING**：警告信息
- **ERROR**：错误信息

### 日志内容

- 阶段开始/结束时间
- 产物文件路径
- 执行耗时
- 错误详情
- 资源使用情况

## 安全性

### 敏感信息处理

- API Token 只从环境变量读取
- 不在日志中输出敏感信息
- Context 对象不包含密码信息

### 错误信息

- 错误信息脱敏处理
- 不暴露系统路径和配置
- 提供通用的错误提示

## 最佳实践

1. **阶段职责单一**：每个阶段只做一件事
2. **产物自包含**：产物文件应包含完整的上下文信息
3. **错误可恢复**：大多数错误应该可以从断点恢复
4. **日志可追溯**：所有操作都应该有日志记录
5. **状态可持久化**：支持从任意检查点恢复执行
